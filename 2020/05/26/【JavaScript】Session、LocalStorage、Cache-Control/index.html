一、课程简介

1.2 Cookie 存在的问题

用户可以随意篡改 Cookie

1.3 Session 与 Cookie 的关系

一般来说，Session 基于 Cookie 来实现。

1.3.1 Cookie

  1. 服务器通过 Set-Cookie 头给客户端一串字符串
  2. 客户端每次访问相同域名的网页时，必须带上这段字符串
  3. 客户端要在一段时间内保存这个Cookie
  4. Cookie 默认在用户关闭页面后就失效，后台代码可以任意设置 Cookie 的过期时间
  5. 大小大概在 4kb（参考资料） 以内

1.3.2 Session（不翻译）

  1. 将 SessionID（随机数）通过 Cookie 发给客户端
  2. 客户端访问服务器时，服务器读取 SessionID
  3. 服务器有一块内存（哈希表）保存了所有 session
  4. 通过 SessionID 我们可以得到对应用户的隐私信息，如 id、email
  5. 这块内存（哈希表）就是服务器上的所有 session

1.3.3 LocalStorage

  1. LocalStorage 跟 HTTP 无关
  2. HTTP 不会带上 LocalStorage 的值
  3. 只有相同域名的页面才能互相读取 LocalStorage（没有同源那么严格）
  4. 每个域名 localStorage 最大存储量为 5Mb 左右（每个浏览器不一样）
  5. 常用场景：记录有没有提示过用户（没有用的信息，不能记录密码）
  6. LocalStorage 永久有效，除非用户清理缓存

1.3.4 SessionStorage（会话存储）

  1. SessionStorage 跟 HTTP 无关
  2. HTTP 不会带上 SessionStorage 的值
  3. 只有相同域名的页面才能互相读取 SessionStorage（没有同源那么严格）
  4. 每个域名 SessionStorage 最大存储量为 5Mb 左右（每个浏览器不一样）
  5. SessionStorage 在用户关闭页面（会话结束）后就失效。

二、 使用Session保存信息

2.1 回忆Cookie课程内容

  • 服务器通过set-Cookie这个请求头给予请求方一个字符串
  • 每次请求相同域名的时候带上Cookie

例子

  • 当用户登录成功， 用户就会获取到一个Cookie，假设Cookie内容为字符串sigh_in_email:1@baidu.com
  • 用户在访问首页的时候，这个请求里面就包含了上面这个字符串Cookie:sigh_in_email:1@baidu.com
  • 服务器读取该用户的Cookie。把该用户的信息的渲染到HTML中返回页面。
  • 如果服务器读取该用户的Cookie发现Cookie为空。则返回所有第一次访问/陌生人的页面。

2.2 Cookie的问题

因为Cookie可以被用户篡改，这是Cookie上的大问题，如果Cookie采用明文展示给用户，用户可以随便改。

2.3 如何解决Cookie篡改问题

怎么解决Cookie篡改问题，只要让用户猜不到就行。

2.3.1 举个例子

  • 比如声明一个空数组session,
  • 在用户登录的时候，如果发现用户曾经注册过，用户账号密码正确时，
  • 就给用户一个Set-Cookie ,sign_in_email=${email}，这就是问题所在。
  • 因为服务器直接把email暴露给了用户，所以我们不要这样写。
  • 而是要创建一个sessionId，seeionId是随机数,
  • 我们不直接把email存储到Cookie里面，而是存储在seeion里面
  • 就是说这个用户登录的email就是服务器找到的对应的email。
  • 然后我们只给它一个sessionId

这样设置的意义

本来用户就是不应该看Cookie内容，现在用户修改Cookie内容也不怕。

怎么利用sessionId

  • 在用户访问首页的时候，服务器去读取一下sessionId里面对应的hash值是什么。

  • 这样我们就知道这个用户sessionId是多少，而email在session里面找。sessionId是随机数

  • 通过sessionId找到hash里面的对象，然后找hash里面的email。这样我们就知道当前用户的email是什么。

    注意：
    只要服务器一关闭，那么这个应用所有的内存就会被释放。内存一释放。session这个hash就为空。就会找不到。 一般来说，服务器是不关闭的，或者在关闭之前，把session存储到一个地方。

2.4 什么是session

什么是session普通理解

一个服务器给每个用户分配了一小块内存，这小块内存就是用户的隐私数据，用户只保存这个内存的key，而这个key是一个随机数。一定要随机数，因为随机数无法猜测。 这就是session。

什么是session面试回答

  • 服务器通过cookie给用户一个sessionId,
  • 然后sessionId对应服务器里面的一小块内存，
  • 每次用户访问服务器的时候，服务器通过sessionId去读取对应的session，然后知道用户的隐私信息

2.5 session能否窜改

不能， 因为sessionId的值是一个随机数。无法猜测。随机数是安全很重要的一部分。

2.6 Cookie的特点（背诵级）

  1. 服务器通过Set-Cookie头给客户端一串字符串
  2. 客户端每次访问相同域名的网页时，必须带上这段字符串
  3. 客户端要在一段时间内保存这个Cookie

2.7 什么是Session

  1. 将 SessionID（随机数）通过 Cookie 发给客户端
  2. 客户端访问服务器时，服务器读取 SessionID
  3. 服务器有一块内存（哈希表）保存了所有 session
  4. 通过 SessionID 我们可以得到对应用户的隐私信息，如 id、email
  5. 这块内存（哈希表）就是服务器上的所有 session

面试官最喜欢为localStorage与Cookie的区别

Cookie的本质是HTTP的一个内容

  1. 服务器通过Set-Cookie头给客户端一串字符串
  2. 客户端每次访问相同域名的网页时，必须带上这段字符串
  3. 客户端要在一段时间内保存这个Cookie

3.1 localStorage是什么?

localStorage是HTML5技术提供的一个API。
window.localstorage

window.localStorage的API

window.localStorage的API有

  • getItem //常用，返回一个函数
  • setItem //常用，返回一个函数
  • removeItem //常用，返回一个函数
  • length
  • clear //一般不使用
  • key

使用console.dir(window.localStorage)可以查看里面API内容

3.2 localStorage的实质

localStorage实质上是一个hash表，localStorage是浏览器上的hash表。

3.3 如何使用localStorage

localStorage只能存储string,其他类型就会转换为string

为什么？因为任何一个对象toString都会变成[object Object]，所以不管存储任意对象，当字符串存储都会变成[object Object]。

使用JSON存储对象

如何解决该问题？使用JSON存储
localStorage.setItem('jsonString',JSON.stringify({name:'obj'}))

JSON.stringify就会将该对象变成JSON字符串

JSON.stringify({name:'obj'})//返回"{"name":"obj"}"

如何读取数据

getItem、setItem、clear它们都是操作当前页面的hash

八、Expires

Expires 响应头包含日期/时间， 即在此时候之后，响应过期。

无效的日期，比如 0, 代表着过去的日期，即该资源已经过期。

如果在Cache-Control响应头设置了 “max-age” 或者 “s-max-age” 指令，那么 Expires 头会被忽略。

语法：Expires:

九、ETag

ETagHTTP响应头是资源的特定版本的标识符。这可以让缓存更高效，并节省带宽，因为如果内容没有改变，Web服务器不需要发送完整的响应。

语法
